# Design — STU-32: Community Registry (search, install, publish)

**Date:** 2026-02-28
**Status:** Approved
**Linear:** https://linear.app/studioag/issue/STU-32

---

## Overview

Build the Studio community registry — a shared package ecosystem where anyone can publish and discover tools, templates, pipelines, integrations, agents, plugins and skills. Two parallel workstreams:

1. **`studio-community` repo** — populate with all existing packages
2. **Studio CLI** — implement `studio registry *` commands

---

## Scope

### In scope (STU-32)
- Migrate 4 project templates from `Studio/cli/templates/projects/` to registry
- Migrate 3 integrations from `Studio/.worktrees/stu-184-integration-plugins/runner/templates/integrations/` (planned alongside STU-184)
- Generate `index.json` via GitHub Action on merge
- Implement all `studio registry` CLI commands
- Refactor `studio init` to use registry (all-or-nothing with the migration)
- Remove bundled templates and integrations from Studio repo after migration

### Out of scope
- Phase 2 API migration (`GET /api/v1/packages`)
- Paid packages, analytics dashboard
- The runner built-in tools (git, repo-manager, search, shell) — these are runtime internals, not registry packages

### `metadata.json` updates

Existing templates are missing the `license` field (required per README). Add `"license": "MIT"` to all official packages. Also strip `pipelines` and `tools_included` fields which are Studio-internal and not part of the registry spec.

Final registry `metadata.json` format:
```json
{
  "name": "software",
  "version": "1.0.0",
  "description": "Code generation with repo, shell and search tools",
  "author": "studio-core",
  "license": "MIT",
  "tags": ["software", "code", "development"],
  "type": "template",
  "studio_version": ">=7.0.0"
}
```

### `index.json` (auto-generated)

Generated by a GitHub Action triggered on every merge to `main`. Do not edit manually.

```json
{
  "generated_at": "2026-02-28T12:00:00Z",
  "version": "1",
  "packages": [
    {
      "name": "software",
      "type": "template",
      "version": "1.0.0",
      "description": "Code generation with repo, shell and search tools",
      "author": "studio-core",
      "tags": ["software", "code", "development"],
      "downloads": 0
    }
  ]
}
```

### GitHub Action: `generate-index.yml`

Triggered on push to `main`. Reads all `*/*/metadata.json`, aggregates into `index.json`, commits back.

## CLI Architecture

### Cache

`~/.studio/registry/index.json` — local cache, TTL 24 hours.

- `search`/`browse` use cache, auto-sync if expired
- `install` triggers sync if cache is expired, then downloads from GitHub
- `studio registry sync` forces a manual refresh

### Commands

```
studio registry search <query> [--type tool|template|pipeline|integration|agent|plugin|skill]
studio registry browse                  # list most popular / recently updated
studio registry install <name>          # install latest
studio registry install <name>@<ver>    # install specific version
studio registry update <name>           # update installed package
studio registry outdated                # list packages with available updates
studio registry remove <name>           # uninstall a package
studio registry publish <path>          # publish via GitHub PR
studio registry audit                   # verify SHA256 integrity of installed packages
studio registry sync                    # force cache refresh
```

### `studio registry install` flow

```
1. Sync cache if expired (fetch index.json from GitHub raw)
2. Resolve package name → metadata (from cache or GitHub API fallback)
3. Download file or directory from:
   https://raw.githubusercontent.com/studio-community/registry/main/<type>/<name>/
4. Copy to destination based on type:
   - tool        → .studio/tools/
   - template    → .studio/projects/<name>/   (+ git init, npm install)
   - pipeline    → .studio/pipelines/
   - integration → .studio/integrations/
   - agent       → .studio/agents/
   - plugin      → .studio/plugins/
   - skill       → .studio/skills/
5. Check requires_binaries — warn if any are missing from $PATH
6. Compute SHA256 checksum of all downloaded files
7. Write to .studio/registry.lock.json
```

### `studio registry publish` flow

```
1. Validate package locally: studio validate <type> <path>
2. Read metadata.json — ensure all required fields present
3. Determine package destination in registry: <type>/<name>/
4. Fork studio-community/registry via GitHub API (skip if fork exists)
5. Create branch: [type]/package-name-vX.Y.Z
6. Copy package files to branch
7. Open PR via GitHub API with title: [type] package-name vX.Y.Z
8. Print PR URL to stdout
```

Requires `gh` CLI authenticated, or `GITHUB_TOKEN` env var.

### Security checks during install

Packages with `execute.type: shell` in `.tool.yaml` or `.integration.yaml` trigger an explicit warning:

```
⚠ This package executes shell commands:
  - run_command: "bash {{input.script}}"
Review the commands above. Install anyway? [y/N]
```

### Lockfile: `.studio/registry.lock.json`

```json
{
  "installed": {
    "software": {
      "version": "1.0.0",
      "type": "template",
      "installed_at": "2026-02-28",
      "sha256": "abc123def456..."
    },
    "linear": {
      "version": "1.2.0",
      "type": "integration",
      "installed_at": "2026-02-28",
      "sha256": "xyz789..."
    }
  }
}
```

## `studio init` refactor

`studio init --template software` becomes a thin wrapper:

```
studio init --template <name> [--name <project-name>]
  → studio registry install <name> (copies to .studio/projects/<project-name>/)
  → git init
  → npm install
  → writes .studio/config.yaml
  → prints "Ready. Run: studio run <name>/feature-builder"
```

The bundled templates in `Studio/cli/templates/projects/` are deleted after this is live. `studio init` without `--template` remains as a blank project scaffolder.
